## 1. Project Overview

JARVIS is an AWS infrastructure repository managed with Terraform and deployed via GitHub Actions. The goal is a clean, repeatable workflow where:

- All infrastructure is defined as code (Terraform) and version-controlled.
- Deployments are manual and environment-aware вЂ” you choose `dev` or `production` when you trigger a deploy.
- AWS resources (EC2, VPC, etc.) are only created or updated when Terraform files actually change вЂ” not on every code push.
- Every push and pull request to `dev` or `main` runs a full CI suite (secret scanning, IaC checks, security scanning, linting).
- Documentation is auto-regenerated by Claude whenever code changes.

**AWS Region:** `eu-north-1` (Stockholm)

## 3. Infrastructure (AWS)

All resources live in `eu-north-1` (Stockholm). The Terraform state is stored remotely in S3, with a separate state file per environment.

### 3.1 State Backend (`terraform/versions.tf`)

```
S3 bucket: jarvis-tf-state-eun1
Region:    eu-north-1
Locking:   use_lockfile = true (native S3 locking, no DynamoDB needed)
Encrypt:   true (AES256)
```

The state key is injected dynamically at `terraform init` time by the deploy workflow:
- Dev:        `env/dev/terraform.tfstate`
- Production: `env/production/terraform.tfstate`

This means dev and production have completely independent state files вЂ” changes to one environment never affect the other.

### 3.2 Bootstrap (`bootstrap/`)

A separate, standalone Terraform project that creates the S3 state bucket before the main Terraform can be used. **Run once** via the Bootstrap GitHub Actions workflow. After this bucket exists, all future `terraform init` calls in `terraform/` will succeed.

Resources created:
| Resource | Name | Purpose |
|---|---|---|
| `aws_s3_bucket` | `jarvis-tf-state-eun1` | Stores all Terraform state files |
| `aws_s3_bucket_versioning` | вЂ” | Keeps history of every state version |
| `aws_s3_bucket_server_side_encryption_configuration` | вЂ” | AES256 encryption at rest |
| `aws_s3_bucket_public_access_block` | вЂ” | Blocks all public access to the bucket |

### 3.3 Main Infrastructure (`terraform/main.tf`)

Every deploy creates or updates the following resources:

#### Networking
| Resource | Value | Purpose |
|---|---|---|
| `aws_vpc` | CIDR `10.0.0.0/16` | Isolated network for the project |
| `aws_internet_gateway` | вЂ” | Connects the VPC to the public internet |
| `aws_subnet` (public) | CIDR `10.0.1.0/24`, AZ `eu-north-1a` | Public subnet where EC2 instances live |
| `aws_route_table` | Route `0.0.0.0/0 в†’ IGW` | Sends all outbound traffic to the internet |
| `aws_route_table_association` | вЂ” | Associates the public subnet with the route table |

#### Security Group
| Rule | Port | Source | Purpose |
|---|---|---|---|
| Ingress | 22 (SSH) | `0.0.0.0/0` | SSH access to EC2 |
| Ingress | 80 (HTTP) | `0.0.0.0/0` | HTTP traffic to the app |
| Egress | All | `0.0.0.0/0` | Unrestricted outbound |

> **Note:** SSH is currently open to the world (0.0.0.0/0). Restrict this to your IP when moving to production.

#### EC2 Instance
| Setting | Dev | Production |
|---|---|---|
| AMI | Amazon Linux 2023 (latest) | Amazon Linux 2023 (latest) |
| Instance type | `t3.micro` | `t3.small` |
| Public IP | Yes (auto-assigned) | Yes (auto-assigned) |
| Name tag | `jarvis-dev` | `jarvis-production` |

The AMI is fetched dynamically at plan time using a `data "aws_ami"` lookup вЂ” always uses the latest Amazon Linux 2023 x86_64 image.

## 4. GitHub Actions Workflows

### 4.1 Bootstrap (`bootstrap.yml`)
**Trigger:** Manual (`workflow_dispatch`)
**When to run:** Once, before any other workflow. Creates the S3 state bucket.

Steps:
1. Checkout code
2. Install Terraform
3. Configure AWS credentials (from `dev` environment secrets)
4. `terraform init` inside `bootstrap/`
5. `terraform apply -auto-approve` вЂ” creates the S3 bucket

**Only run this once.** Running it again is safe (Terraform is idempotent) but unnecessary.

---

### 4.2 Deploy (`deploy.yml`)
**Trigger:** Manual (`workflow_dispatch`) вЂ” you select `dev` or `production`
**Purpose:** Deploys infrastructure and/or app code to the chosen environment

The workflow has three jobs:

#### Job 1: `changes` вЂ” Detect what changed
Compares `HEAD` vs `HEAD~1` to see which files changed. Sets two output flags:
- `infrastructure=true` if any `.tf` files changed under `terraform/` or `bootstrap/`
- `app=true` if any files outside of `terraform/`, `bootstrap/`, `.github/`, or `docs/` changed

#### Job 2: `infrastructure` вЂ” Terraform (only runs if `infrastructure=true`)
1. Checkout code
2. Install Terraform `~> 1.15`
3. Configure AWS credentials from GitHub environment secrets
4. `terraform init` with the per-environment state key
5. `terraform plan` using the environment's `.tfvars` file
6. `terraform apply` the saved plan
7. **Smoke test 1:** Re-run `terraform plan` вЂ” exit code 0 means no drift
8. **Smoke test 2:** Run `aws-health.sh` вЂ” verifies EC2 is in "running" state
9. **Smoke test 3:** Run `http-checks.sh` вЂ” verifies HTTP endpoints (placeholder)

#### Job 3: `app` вЂ” App deploy (only runs if `app=true`)
Currently a placeholder. Add steps here when you have application code to deploy (e.g. SCP to EC2, push Docker image, upload to S3).

**Trigger:** Push to `dev` or `main` (ignores changes to `documentation.md` itself to avoid loops)

## 5. CI Pipeline & Security Checks

All five jobs must pass before any PR can merge to `main` (enforced via branch protection rules).

| # | Job | Tool | What it checks | Fails hard? |
|---|---|---|---|---|
| 1 | Secret scan | gitleaks v8.18.4 | Secrets, API keys, tokens accidentally committed | Yes |
| 2 | Terraform checks | terraform fmt + validate | HCL formatting and syntax validity | Yes |
| 3 | Terraform security | tfsec | AWS security misconfigurations (open ports, missing encryption, etc.) | No (soft fail) |
| 4 | Python lint | pylint (score в‰Ґ 8.0) | Code quality, unused imports, naming conventions | Yes |
| 5 | Python security | bandit (-ll) | Security bugs: eval, subprocess injection, weak crypto, hardcoded secrets | Yes (medium+) |

### gitleaks config (`.gitleaks.toml`)
Extends the default ruleset. Allowlisted safe strings:
- `jarvis-tf-state-eun1` вЂ” S3 bucket name, not a secret
- `eu-north-1`, `us-east-1` вЂ” AWS region strings

Ignored paths:
- `terraform/.terraform.lock.hcl` вЂ” provider lock file
- `documentation.md` вЂ” auto-generated, may contain code snippets

### pylint config (`pyproject.toml`)
- Python 3.12
- Max line length: 100
- Disabled: `C0114` (module docstrings), `C0116` (function docstrings), `W0511` (TODO comments)

### Dependabot (`.github/dependabot.yml`)
Runs every Monday and opens PRs for:
- Outdated or vulnerable **pip packages** (e.g. `anthropic` SDK)
- Outdated **GitHub Actions** versions (e.g. `actions/checkout`, `hashicorp/setup-terraform`)

## 6. Deployment Guide

### First-time setup (run once)

1. **Add GitHub environment secrets**
   - Go to Settings в†’ Environments в†’ create `dev` and `production`
   - Add to each: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`

2. **Add repository secrets**
   - Go to Settings в†’ Secrets and variables в†’ Actions
   - Add: `ANTHROPIC_API_KEY` (for the docs workflow)

3. **Run Bootstrap workflow**
   - GitHub в†’ Actions в†’ "Bootstrap (run once)" в†’ Run workflow
   - This creates the `jarvis-tf-state-eun1` S3 bucket in `eu-north-1`

### Normal development flow

```
1. Write code on dev branch
2. git push origin dev
   в†’ CI runs automatically (gitleaks, terraform, tfsec, pylint, bandit)
   в†’ docs workflow runs в†’ documentation.md auto-updated

3. Deploy to dev (to test infrastructure changes):
   GitHub в†’ Actions в†’ Deploy в†’ Run workflow в†’ select: dev

4. Verify everything works in dev

5. Open PR: dev в†’ main
   в†’ CI must pass all 5 checks before merge is allowed

6. Merge PR to main

7. Deploy to production:
   GitHub в†’ Actions в†’ Deploy в†’ Run workflow в†’ select: production
```

### When does Terraform actually run?

Only when `.tf` files under `terraform/` or `bootstrap/` have changed in the last commit. If you only changed Python scripts, documentation, or workflow files вЂ” the infrastructure job is skipped entirely.

## 8. Smoke Tests

Run automatically after every successful `terraform apply` in the deploy workflow.

### `aws-health.sh`
1. Calls `aws sts get-caller-identity` to verify credentials are working
2. Reads `instance_id` from Terraform outputs
3. Calls `aws ec2 describe-instances` and checks the state is `"running"`
4. Fails with exit code 1 if the instance is not running

### `http-checks.sh`
Currently a placeholder. Add checks here as you deploy HTTP services. Example pattern (already in the file):
```bash
ENDPOINT=$(terraform -chdir=terraform output -raw api_endpoint)
STATUS=$(curl -sf -o /dev/null -w "%{http_code}" --max-time 10 "$ENDPOINT/health")
[[ "$STATUS" == "200" ]] || exit 1
```

## 9. Documentation Auto-Generation

The `generate_docs.py` script collects these file patterns and sends them to Claude:
- `terraform/**/*.tf`
- `bootstrap/**/*.tf`
- `.github/workflows/*.yml`
- `.github/smoke-tests/*.sh`

It uses **prompt caching** (`cache_control: ephemeral`) to avoid re-processing the same file content on repeated runs вЂ” reduces cost and latency when only one file changed.

Model: `claude-sonnet-4-6` | Max tokens: `4096`
